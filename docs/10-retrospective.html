1 プロジェクト振り返り

企画・設計段階の振り返り 必須
マーケットリサーチ、ペルソナ設計、情報設計について

企画・設計段階では、生成AIに着手する前に企画提案書の精度向上へ十分な工数を投下したことが、後続工程の安定化に大きく寄与した。
初期段階で目的、構成、期待成果を整理できていたため、仕様策定や設計判断の迷いが減り、全体として手戻りの少ない進行を実現できた。
また、サイトマップやデザインガイドラインはAIを活用して整合性を確認しながら作成したことで、情報構造と表現方針の両面で一貫性の高い設計にまとめられた。
一方で、曖昧なPromptによる指示は成果物の品質を不安定にし、途中での修正・巻き戻し対応に工数を要した。
次回は、AI活用時の指示テンプレートと中間レビューの運用を事前に組み込み、品質と進行管理の両立をさらに強化したい。

デザイン段階の振り返り 必須
デザインガイドライン、ワイヤーフレーム、Figma作業について

デザイン段階では、AIを活用することで配色、余白、情報配置など複数の案を短時間で比較でき、初期検討のスピードを大きく高めることができた。
特に、サイト全体のトンマナや画面間の整合性を早い段階で確認できたことは、後続の実装負荷軽減にもつながった。
一方で、AIが生成する案は一見完成度が高く見えても、目的や利用者像に十分即していない場合があり、そのまま採用すると表層的なデザインに偏るリスクがあることも認識できた。
次回は、評価基準を先に定義したうえでAIを発想支援と整理支援に位置づけ、人の判断による選定と改善をより明確に分離して進めたい。

フロントエンド実装の振り返り 必須
HTML/CSS/JavaScript の実装について

フロントエンド実装段階では、HTML/CSS/JavaScriptを役割ごとに整理しながら進めたことで、画面構造、見た目、挙動の責務を切り分けて実装できた。
また、AIを活用して実装方針や記述案を早い段階で比較できたことで、初動の迷いを減らし、コーディングの効率向上につなげることができた。
特に、設計段階で整理した情報構造やデザイン方針が明確だったため、AIの提案内容も取捨選択しやすく、全体として一貫性のある実装を進めやすかった。
一方で、見た目や動作を優先して細部を都度調整する場面では、CSSやJavaScriptの記述が局所最適になりやすく、保守性を意識した構成の重要性を再認識した。
次回は、AIを補助的なレビューと発想支援に位置づけつつ、命名規則やスタイル設計の方針を早い段階で固め、修正に強いフロントエンド実装を目指したい。

バックエンド実装の振り返り 必須
Java/Spring Boot の実装について

バックエンド実装段階では、Supabaseと連携しつつ、最終的にGitHub Pagesで公開できる構成を成立させる必要があり、一般的なサーバーサイド開発とは異なる制約を強く意識して進める必要があった。
その中で、認証やデータ連携の実現方法、公開環境に適した構成案の整理にAIを活用したことで、複数の選択肢を比較しながら現実的な実装方針を早期に固めることができた。
特に、利用環境の制約を前提として役割分担を整理できたことで、Supabaseを活用した機能実装と静的公開の両立に見通しを持って進められた点は大きな成果だった。
一方で、構成が複雑になるほど実装上の前提条件や依存関係の確認が重要となり、AIの提案もそのまま採用するのではなく、技術的妥当性を都度検証する必要があると感じた。
次回は、公開環境の制約とバックエンド要件をより早い段階で整理したうえで、AIを設計レビューと代替案検討に活用し、より安定した構成で開発を進めたい。

AIの活用によって根底から変わるシステム開発のあり方

1. はじめに（何をしたかったか）

本記事では、AIの活用によってシステム開発の進め方がどのように変わるのかを、初学者向けに整理する。従来の開発では、人が要件整理、設計、実装、テストを順に積み上げていく形が一般的だった。
しかし現在は、AIを活用することで、調査、たたき台作成、コード例の生成、レビュー補助までを高速に進められるようになった。
私はこの変化を、開発現場における一種のBig Bangだと感じている。
重要なのは、AIが人の仕事をそのまま奪うという見方ではなく、人とAIの役割分担が変わり、それぞれの得意分野で共存する時代になったと理解することである。

2. 課題（何が難しかったか）

実際にAIを使ってみると、単に質問すれば正しい答えが返るわけではなかった。
指示が曖昧だと、もっともらしく見えても目的に合わない提案が返ることがある。
また、環境制約や仕様条件を十分に伝えないと、実際には採用できないコードが出力されることもあった。
つまり難しかったのは、AIを使うこと自体ではなく、AIに何を、どこまで、どの粒度で伝えるかを人間側が設計することだった。

3. 調査（どのように調べたか）

私はまず、AIにいきなり実装させるのではなく、要件、制約、目的を箇条書きで整理してから質問するようにした。
たとえば「GitHub Pagesで公開する」「Supabaseと連携する」「初学者でも理解できる構成にする」といった前提条件を先に明示した。
そのうえで、1回で完成を目指すのではなく、「構成案を出す」「候補を比較する」「不明点を絞る」「最後に実装する」という流れで段階的に調査した。
この方法により、AIを検索代替ではなく、整理・比較・発想支援の相手として使えることが分かった。

4. 実装（コード付きで解説）

たとえば、フロントエンドからSupabaseへデータを登録する処理を考えるときも、AIに完成品を丸投げするのではなく、必要な処理単位で相談すると理解しやすい。
以下は、入力値を取得して登録するシンプルな例である。 

async function saveMessage() {
  const name = document.getElementById('name').value.trim();
  const message = document.getElementById('message').value.trim();

  if (!name || !message) {
    alert('入力内容を確認してください。');
    return;
  }

  const { error } = await supabase
    .from('messages')
    .insert([{ name, message }]);

  if (error) {
    console.error(error);
    alert('保存に失敗しました。');
    return;
  }

  alert('保存しました。');
}

ここで大切なのは、AIに「このコードを書いて」で終わるのではなく、「なぜ入力チェックが必要か」「なぜエラー処理が必要か」まで説明させることだ。
AIはたたき台の生成が得意であり、人はそのコードが本当に要件に合っているかを判断する役割を持つ。
この分担によって、学習速度も実装速度も高めやすくなる。

5. 結果（どう解決したか）

最終的に、AIをうまく活用するには、全部を任せるのではなく、役割を分けることが重要だと分かった。
AIは、案出し、調査の整理、コードのひな形作成、レビュー補助に強い。
一方、人は目的設定、優先順位付け、正誤判断、最終意思決定に強い。
この分担を意識したことで、手戻りを減らしながら、限られた時間でも開発を前に進めやすくなった。
AIによるBig Bangとは、開発者が不要になることではなく、開発者の役割が「書く人」から「考え、選び、統合する人」へ広がることだと捉えている。

6. 学び（何を得たか）

この経験から得た最大の学びは、これからのシステム開発では、コードを書く力だけでなく、AIを適切に使いこなす力が重要になるということである。
初学者にとっても悲観する必要はなく、むしろ基礎を学びながらAIを補助役として使うことで、理解を深めやすくなる。
AIは人の職を奪う存在というより、人の役割を再定義する存在である。
だからこそ、AIに任せる部分と自分が責任を持つ部分を切り分け、共存の前提で活用する姿勢が今後ますます重要になる。

1. はじめに（何をしたかったか）

本記事では、AIと一緒に考えながら実装したデジタル調香アプリについて紹介する。
このアプリのコンセプトは、利用者がディスプレイ上で香りのNotesと配合を調整し、その結果を色とサウンドで表現することで、香りを視覚と聴覚でも体験できるようにすることである。
本来、香りは実際に嗅がなければ分かりにくいが、このアプリでは操作と演出を組み合わせることで、香りの印象をデジタル上でも感じ取れる体験を目指した。
私が実現したかったのは、単なる情報表示のアプリではなく、利用者が自分で組み合わせを試しながら、感覚的に楽しめるインタラクティブな仕組みである。
その実装を進める中で、AIはコードを作る道具というより、体験設計や処理の整理を一緒に考える相手として大きな役割を果たした。

2. 課題（何が難しかったか）

最も難しかったのは、香りという目に見えないものを、画面上の操作だけでどう伝えるかを考えることだった。
Notesや配合比率の変更によって印象が変わることを、利用者に分かりやすく伝える必要があったため、単に数値を表示するだけでは不十分だった。
そこで、香りのイメージを色の変化やサウンド表現に置き換える必要があったが、どの要素をどう結び付ければ自然に感じられるかを整理するのに苦労した。
また、実装面でも画面上の操作、表示の変化、データ管理をどう組み合わせるかを考える必要があり、見た目の面白さと処理の分かりやすさの両立が課題だった。
一部では外部サービスとの連携や公開環境も考慮したが、それ以上に難しかったのは、香りの体験そのものをどうデジタルに翻訳するかだった。

3. 調査（どのように調べたか）

調査では、まず「香りをどうすれば画面上で体験として伝えられるか」を軸に考えた。
そのために、Notesごとの印象を言葉で整理し、それぞれに対応しそうな色や音のイメージを洗い出した。
さらに、AIに対して「この香りの印象を色で表すならどうなるか」「この配合変化を体験として見せるには何が自然か」といった形で問いかけ、発想の候補を広げた。
ここで重要だったのは、AIに完成形を求めるのではなく、アイデアの比較相手として使うことだった。
その結果、人だけでは思いつきにくい表現案も得られ、体験設計の方向性を整理しやすくなった。

4. 実装（コード付きで解説）

実装では、利用者が選んだNotesや配合に応じて、画面上の色や音の印象が変わる流れを意識した。
以下は、選択された配合情報をもとに画面表示を更新するイメージを簡略化した例である。
コメントを入れることで、初学者でも処理の役割を追いやすくしている。 

function updateBlendExperience(noteName, ratio, color, soundLabel) {
  // 選択中のNotes名を画面に表示する
  document.getElementById('selected-note').textContent = noteName;

  // 配合比率を利用者に分かりやすく表示する
  document.getElementById('selected-ratio').textContent = ratio + '%';

  // 香りの印象を色として背景に反映する
  document.getElementById('experience-panel').style.backgroundColor = color;

  // サウンドのイメージをテキストで伝える
  document.getElementById('sound-description').textContent = soundLabel;
}

この処理自体は難しい構文を多く使っているわけではないが、「どの情報を変えると利用者が体験の変化を感じられるか」を考えることが重要だった。
たとえば、Notes名だけを表示しても体験としては弱いため、色や音の説明も合わせて変化させることで、香りのイメージが伝わりやすくなる。
AIは、このような処理のたたき台や整理案を出すのに役立ったが、最終的に何を採用するかは、自分でアプリの目的に照らして判断した。
必要に応じてデータ保存や公開構成も調整したが、実装の中心はあくまで「香りの体験をどう成立させるか」にあった。

5. 結果（どう解決したか）

最終的には、利用者がNotesと配合を操作すると、その変化が色とサウンドのイメージとして返ってくる流れを形にすることで、デジタル調香というコンセプトを画面上で表現できた。
これにより、香りそのものを直接届けられなくても、香りの印象や変化を感覚的に楽しめる体験に近づけることができた。
また、AIと対話しながら考えたことで、発想を広げる段階と、実現可能な形に絞り込む段階を分けて進めやすくなった。
その結果、AIは人の代わりに完成品を作る存在というより、人の考えを整理し、表現の選択肢を広げる存在だと実感した。
この点は、これからの開発において非常に大きな変化だと感じている。

6. 学び（何を得たか）

この経験を通じて学んだのは、これからのシステム開発では、機能を作る力だけでなく、体験をどう設計するかを考える力がより重要になるということである。
特にAIを活用すると、実装案を早く出したり、発想を広げたりすることはできるが、何を良い体験とするかを決めるのは人の役割である。
AIは人の職を奪うものというより、人が担うべき役割を変える存在だと感じた。
人は目的を定め、利用者体験を考え、最終判断を行う。
AIは整理、比較、たたき台作成を支える。
この役割分担を理解して活用することが、今後の開発ではますます重要になると考えている。
