1 システム概要

システムの目的 必須
このシステムが解決する課題と提供する価値

本システムは、`GitHub Pages + HTML/CSS/JavaScript + Supabase` を採用し、店舗管理者・スタッフがワークショップ予約、掲載コンテンツ、商品・店舗情報、顧客対応を一元管理できるようにする運用管理システムである。

公開サイトは静的HTMLとして低コストで配信しつつ、管理画面から Supabase の認証・データベース・ストレージを通じて内容を更新することで、サイト更新と効果測定を迅速化する。これにより、手作業を減らし、体験価値を正しく届け、予約と売上の回復を支える。

項目 内容
誰が使うか 店舗管理者・スタッフ・Web担当者が、日常運用と掲載更新、問い合わせ対応のために利用
解決する課題 予約・店舗・商品・顧客・掲載素材の情報が分散し、更新反映の遅れ、ダブルブッキング、問い合わせ対応漏れ、効果検証の遅れが発生している
できること ログイン認証／ダッシュボードでの状況把握／トップ編集・導線設定・カード管理・記事管理／商品・店舗・予約・問い合わせの管理／画像・音源のアセット管理／KPI確認
提供価値 静的サイトの軽さとBaaSの更新性を両立し、低コストで運用開始できる。業務フロー統合で手作業とエラーを減らし、改善サイクルを短縮して予約・売上回復と運用コスト抑制を両立する
採用構成 公開サイト・管理画面は GitHub Pages で静的配信し、Supabase の Auth / Database / Storage を JavaScript から利用する。将来的に必要となれば Spring Boot / Java に段階的に移行可能
コスト観点 初期は GitHub Pages と Supabase の無料枠中心で小さく開始し、利用量や業務要件の増加に合わせて Edge Functions や Spring Boot の導入を検討する
システム構成図 必須
フロントエンド ↔ バックエンド ↔ データベースの関係を図示

採用案は「GitHub Pages で静的HTMLを公開し、JavaScript から Supabase を利用して動的データを扱う」構成とする。静的配信の軽さと低コストを維持しつつ、ログイン、CRUD、アセット管理、ダッシュボード表示を実現する。

利用者
管理者 / スタッフ PC・タブレットから管理画面を利用
HTTPS
一般ユーザー 公開サイトを閲覧し予約導線へ進む
フロントエンド
GitHub Pages 静的HTML / CSS / JavaScript を公開
公開サイト トップ / 下層ページ / 予約導線
管理画面 ログイン / ダッシュボード / 各管理画面
BaaS / データ層
Supabase Auth ログイン認証 / セッション管理 / 権限制御
Supabase Database コンテンツ / 商品 / 店舗 / 予約 / 顧客 / KPI 用データ
Supabase Storage 画像 / 音源 / メディアアセット
将来拡張
Supabase Edge Functions 通知、権限昇格、重い処理を追加可能
Spring Boot / Java 複雑な業務ロジックや外部連携が増えた場合に API 層として追加可能

運用フロー

  1. 管理者が GitHub Pages 上の管理画面にアクセスし、Supabase Auth でログインする。
  2. 各管理画面で文言・画像・商品・店舗・予約情報を更新する。
  3. 更新データは Supabase Database / Storage に保存される。
  4. 公開サイト側の JavaScript がデータを読み込み、表示内容へ反映する。

コストと拡張性

  • 初期は GitHub Pages と Supabase の無料枠中心で開始しやすい。
  • 静的配信のため、公開サイト側のホスティングコストを抑えやすい。
  • 将来、承認フローや外部連携が必要になれば Spring Boot / Java を後付けできる。

4 機能詳細仕様

Must機能を優先して詳細仕様化する 本節では `F-001 / F-002 / F-003 / F-008 / F-009` を対象に、画面レイアウト・入力項目・処理フロー・正常時/異常時・遷移先を整理する。Should / Could は後続で追加する。

F-001 ログイン・権限管理

機能名：ログイン・権限管理 必須
機能ID：F-001

1. 画面レイアウト
ログイン画面は中央にフォームを配置し、上から「メールアドレス」「パスワード」「ログイン状態を保持」「ログイン」ボタンを縦並びで配置する。フォーム下部に「パスワード再設定」リンクとエラーメッセージ領域を置き、ヘッダーやサイドメニューは表示しない。ログイン成功後のダッシュボードは、上部にユーザー情報とログアウト、左側にメニュー、中央に KPIカード・優先タスク・主要導線ショートカットを配置する。

3. 処理フロー
ユーザーがメールアドレスとパスワードを入力して `ログイン` を押下 → フロントエンドが入力値を検証 → Supabase Auth へ認証要求 → 認証成功時にセッションを取得しロール情報を読込 → 権限に応じたメニューを生成 → ダッシュボードで予約数、CTAクリック率、未対応問い合わせ数を取得して表示する。

4. 正常時の動作
認証成功後、ダッシュボードに遷移し、ログインユーザーの権限に応じて利用可能なメニューのみ表示する。KPIカードと優先タスクが最新状態で表示され、各管理画面への移動起点として機能する。ログイン状態保持が有効な場合はブラウザセッションを継続する。

5. エラー時の動作
必須未入力や形式不正は項目下に即時表示する。認証失敗時はフォーム上部に「メールアドレスまたはパスワードが正しくありません」を表示する。Supabase 接続失敗時は「現在ログインできません。時間をおいて再試行してください」を表示し、再送ボタンを有効のまま維持する。

6. 画面遷移先
成功時：ダッシュボード
失敗時：ログイン画面に留まる
補助遷移：パスワード再設定フロー

入力項目一覧 必須
項目名 データ型 必須 バリデーション 備考
メールアドレス String ○ 必須 / メール形式 / 255文字以内 Supabase Auth のログインID
パスワード Password String ○ 必須 / 8〜72文字 / 前後空白除去 入力値は画面上マスク表示
ログイン状態を保持 Boolean — 任意 セッション保持可否を指定

F-002 トップヒーロー/スライダー管理

機能名：トップヒーロー/スライダー管理 必須
機能ID：F-002

1. 画面レイアウト
左側にサイドメニュー、上部にパンくずと保存状態を表示する。メインエリア左カラムに編集フォーム、右カラムに公開サイトプレビューを配置する。フォームには「見出し」「リード文」「ボタン文言」「リンク先」「ヒーロー画像」「表示順」を並べる。

3. 処理フロー
担当者がトップ編集画面を開く → 現在公開中のヒーロー設定を Supabase Database / Storage から取得 → 項目を編集し `保存` を押下 → バリデーション後にレコード更新 → 公開サイト側の取得データが更新され、即時反映される。

4. 正常時の動作
保存後に成功メッセージを表示し、更新時刻・更新者を画面上部へ反映する。プレビューも最新内容に更新され、公開サイト側でも同内容が取得される。

5. エラー時の動作
未入力や文字数超過は項目単位で表示する。画像未選択で保存した場合は既存画像を維持する。Storage アップロード失敗時はレコード更新を中断し、「画像の保存に失敗しました」と表示する。DB更新失敗時はフォーム値を保持したまま再試行可能とする。

6. 画面遷移先
保存後：トップ編集画面に留まる
補助遷移：アセット管理、ダッシュボード

入力項目一覧 必須
項目名 データ型 必須 バリデーション 備考
見出し String ○ 必須 / 60文字以内 トップFVの主コピー
リード文 Text ○ 必須 / 160文字以内 補足説明文
ボタン文言 String ○ 必須 / 20文字以内 CTAラベル
リンク先URL String ○ 必須 / 相対URLまたはHTTPS URL形式 公開サイト内導線を想定
ヒーロー画像 File / String — jpg/png/webp / 5MB以内 未変更時は既存画像を継続利用
表示順 Integer ○ 必須 / 1以上 / 重複時は保存不可 スライダー時の順序管理

F-003 体験導線設定

機能名：体験導線設定（デジタル調香・ワークショップ） 必須
機能ID：F-003

1. 画面レイアウト
上部に画面タイトルと保存ボタンを配置する。中央に導線一覧テーブルを置き、各行に「ステップ番号」「表示名」「リンク先」「表示状態」「並び替え操作」を表示する。下部に簡易プレビューと更新履歴を配置する。

3. 処理フロー
担当者が対象ステップを選択 → 表示名やリンク先、順序を編集 → `保存` を押下 → 順序重複やURL形式を検証 → Supabase Database を更新 → 公開サイトが新導線設定を取得し即時反映する。

4. 正常時の動作
保存完了メッセージを表示し、導線一覧とプレビューを最新内容へ更新する。公開サイト上のCTA文言、リンク先、表示順が更新される。

5. エラー時の動作
URL形式不正、表示名未入力、順序重複は行単位でエラー表示する。更新処理失敗時は保存前のサーバーデータを保持しつつ、編集内容を画面に残す。接続エラー時は「反映に失敗しました」を表示し再試行を促す。

6. 画面遷移先
保存後：導線設定画面に留まる
補助遷移：ダッシュボード、カード管理、商品管理

入力項目一覧 必須
項目名 データ型 必須 バリデーション 備考
ステップ番号 Integer ○ 必須 / 1以上 / 重複不可 表示順制御に使用
表示名 String ○ 必須 / 30文字以内 CTAや導線ラベル
リンク先URL String ○ 必須 / 相対URLまたはHTTPS URL形式 予約ページ・体験ページ等
補足文言 String — 80文字以内 任意の説明テキスト
表示状態 Boolean / Enum ○ 公開 / 非表示 非表示時は公開サイトへ出さない

F-008 ユーザー管理

機能名：ユーザー管理 必須
機能ID：F-008

1. 画面レイアウト
上部に検索バー、会員種別フィルタ、状態フィルタを配置する。中央にユーザー一覧テーブルを置き、列は「氏名」「メールアドレス」「会員種別」「状態」「最終利用日」「操作」とする。右側またはモーダルで詳細/編集パネルを表示する。

3. 処理フロー
管理者がユーザー管理画面を開く → 一覧を取得 → 検索または絞り込みで対象ユーザーを選択 → `詳細` または `編集` を開く → 利用停止やメモ更新を実行 → Supabase Database を更新し一覧へ反映する。強権限変更が必要な場合は拡張機能へ委譲する。

4. 正常時の動作
詳細情報が表示され、停止状態や内部メモが更新される。更新後は一覧の状態表示と更新日時が最新化される。予約履歴リンクから予約・問い合わせ管理へ遷移できる。

5. エラー時の動作
検索条件不備は即時補正し、0件時は「該当ユーザーなし」を表示する。更新権限不足時は「この操作は許可されていません」を表示する。強権限変更が必要な操作は実行せず、将来拡張対象として案内する。

6. 画面遷移先
詳細/編集後：ユーザー管理画面に留まる
補助遷移：予約・問い合わせ管理、ダッシュボード

F-009 予約/問い合わせログ管理

機能名：予約/問い合わせログ管理 必須
機能ID：F-009

1. 画面レイアウト
上部に「予約 / 問い合わせ」タブ、検索条件、ステータスフィルタ、CSV出力ボタンを配置する。中央に案件一覧を表示し、選択した行の詳細情報を下部または右側パネルに表示する。詳細には顧客情報、対象店舗、希望日時、問い合わせ本文、対応履歴を含める。

3. 処理フロー
担当者が一覧を開く → 未対応案件や日時条件で絞り込む → 対象案件の `詳細` を開く → `ステータス更新` で対応中・完了・キャンセル等に変更 → 更新結果を Supabase Database に保存 → 一覧・ダッシュボードの件数を更新する。

4. 正常時の動作
更新後、一覧のステータスと更新日時が即時反映される。必要に応じて対象ユーザーや店舗詳細へ遷移し、関連情報を連続確認できる。CSV出力時は現在の検索条件に一致するデータをダウンロードする。

5. エラー時の動作
検索条件不正は入力欄にエラー表示する。ステータス更新失敗時は行の変更をロールバックし、「保存に失敗しました」を表示する。CSV生成失敗時は画面遷移せず再試行を案内する。

6. 画面遷移先
詳細/更新後：予約・問い合わせ管理画面に留まる
補助遷移：ユーザー管理、店舗管理、ダッシュボード

5 非機能要件

`GitHub Pages + HTML/CSS/JavaScript + Supabase` 前提で非機能要件を定義する 低コスト運用を前提とするため、セキュリティ・可用性・障害時の案内を Must とし、Should / Good to have は優先度順で整理する。
セキュリティ要件 必須
認証・認可・データ保護に関する要件
優先度 要件 内容 補足
Must 認証 管理画面は Supabase Auth によるログイン認証を必須とし、未認証状態では管理画面へアクセスできないこと。 メールアドレス + パスワード認証を基本とする。
Must 認可 Supabase の RLS を利用し、ロールごとに参照・更新可能なデータ範囲を制御すること。 一般ユーザー向けデータと管理画面向けデータを分離する。
Must 鍵管理 クライアント側では `anon key` のみ利用し、`service_role` や秘密情報を静的HTML/JSへ含めないこと。 強権限操作は将来 Edge Functions 等へ分離する。
Must 通信保護 公開サイト・管理画面・Supabase 間の通信は HTTPS を前提とすること。 外部リンクや画像URLも HTTPS を優先する。
Must 入力値検証 フォーム入力値はフロント側で形式検証し、保存前に必須・型・長さ・URL形式・ファイル形式を確認すること。 XSS を防ぐため、HTMLとして解釈しない文字列保存を基本とする。
Should-1 アクセス制御強化 一定回数以上のログイン失敗時に再試行間隔を空ける、または案内を強化すること。 不正試行の抑制策。
Should-2 監査証跡 ログイン、保存、ステータス更新、利用停止操作を記録すること。 最低限は日時・操作者・対象IDを残す。
Good 二段階認証 将来的に管理者アカウントへ多要素認証を導入できる設計とする。 運用負荷と導入タイミングを見て判断する。

将来、Spring Boot / Java を導入して API 層を追加する場合の補足セキュリティ要件

項目 要件 補足
認証方式（Spring Security） Spring Security を利用し、管理画面/API へのアクセスを認証必須とする。 セッション認証または JWT ベース認証を採用し、管理者向けAPIを保護する。
パスワードの暗号化方式 パスワードは平文保存を禁止し、`BCrypt` 等の一方向ハッシュで保存する。 強度変更に対応しやすい PasswordEncoder を使用する。
CSRF対策 フォームベース認証を採用する場合は CSRF トークンを有効化する。 API をステートレス運用する場合も、方式に応じた保護方針を明確にする。
XSS対策 入力値のサニタイズと出力時エスケープを徹底し、HTMLをそのまま描画しない。 テンプレート描画・JSON応答の双方で対策を行う。
SQLインジェクション対策 プレースホルダ付きクエリ、ORM、Repository 層を利用し、文字列連結SQLを禁止する。 動的検索条件もバインド変数で処理する。
セッション管理 セッションタイムアウト、同時ログイン方針、ログアウト時の無効化を定義する。 管理画面では一定時間無操作時に再認証を求めることを推奨する。
