1 ER図

ER図 必須
テーブル間のリレーション図（テキストベースまたは画像で記載）

採用構成は `GitHub Pages + JavaScript + Supabase` を前提とするため、認証は Supabase Auth の `auth.users` を利用し、業務データはアプリ側テーブルで管理する。以下は Must 機能を中心に、将来拡張で参照される主要テーブルも含めた ER 図である。

リレーション 種類 意味
`auth.users` → `user_profiles` 1対1 Supabase Auth の認証主体に対して、アプリ側プロフィールを1件対応させる。
`roles` ↔ `user_profiles` 多対多 `user_role_assignments` を介して、将来の複数権限付与に対応する。
`content_assets` → `top_hero_items` 1対多 1つのアセットを複数のヒーロー/掲載要素から再利用できる。
`user_profiles` → `top_hero_items` / `journey_steps` 1対多 誰が更新したかを保持し、運用履歴を追跡できる。
`user_profiles` → `reservations` / `inquiries` 1対多 1人の顧客が複数の予約・問い合わせを持てる。
`stores` → `reservations` 1対多 1店舗に複数予約が紐づく。店舗管理機能拡張時も整合する。
`reservations` / `inquiries` → 各ステータス履歴 1対多 対応状況の変更履歴を監査目的で残す。

補足: `auth.users` は Supabase 管理テーブルであり、通常の業務カラム追加は `user_profiles` 側で行う。`stores` は現時点では Could 機能だが、予約ログの参照整合性を保つため ER 図に含めている。

auth.users uq_auth_users_email email UNIQUE ログインIDの一意性確保と認証時検索の高速化
user_profiles uq_user_profiles_auth_user_id auth_user_id UNIQUE `auth.users` との1対1対応を保証するため
user_profiles idx_user_profiles_display_name display_name INDEX ユーザー名検索や候補表示に使用するため
user_profiles idx_user_profiles_account_status account_status INDEX 利用状態での絞り込みに使用するため
roles uq_roles_role_code role_code UNIQUE ロールコードの重複を防ぎ、権限参照を高速化するため
user_role_assignments idx_user_role_assignments_user_profile_id user_profile_id INDEX ユーザーごとのロール取得に使用するため
user_role_assignments idx_user_role_assignments_role_id role_id INDEX ロール単位でのユーザー参照に使用するため
user_role_assignments uq_user_role_assignments_user_role user_profile_id, role_id UNIQUE 同一ユーザーへの同一ロール重複付与を防ぐため
