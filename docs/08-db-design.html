1 ER図

ER図 必須
テーブル間のリレーション図（テキストベースまたは画像で記載）

採用構成は `GitHub Pages + JavaScript + Supabase` を前提とするため、認証は Supabase Auth の `auth.users` を利用し、業務データはアプリ側テーブルで管理する。以下は Must 機能を中心に、将来拡張で参照される主要テーブルも含めた ER 図である。

[auth.users] 1 ──── 1 [user_profiles]
  id (PK)               id (PK)
  email                 auth_user_id (FK, UNIQUE)
  encrypted_password    display_name
  last_sign_in_at       account_status
                         last_login_at
                         created_at / updated_at

[roles] 1 ──── * [user_role_assignments] * ──── 1 [user_profiles]
  id (PK)                id (PK)                    id (PK)
  role_code              user_profile_id (FK)       auth_user_id (FK)
  role_name              role_id (FK)               display_name
  description            assigned_at                account_status

[content_assets] 1 ──── * [top_hero_items]
  id (PK)                 id (PK)
  file_path               title
  file_type               lead_text
  alt_text                cta_label
  bucket_name             cta_url
  uploaded_by (FK)        asset_id (FK)
  created_at / updated_at display_order
                          is_active
                          updated_by (FK)

[user_profiles] 1 ──── * [top_hero_items]
  id (PK)                 updated_by (FK)
  display_name            ...

[user_profiles] 1 ──── * [journey_steps]
  id (PK)                 id (PK)
  display_name            step_no
                          step_name
                          link_url
                          helper_text
                          is_visible
                          updated_by (FK)
                          created_at / updated_at

[user_profiles] 1 ──── * [reservations] * ──── 1 [stores]

  display_name            customer_profile_id (FK) store_name
                          store_id (FK)            prefecture
                          reservation_type         address
                          reserved_at              is_active
                          participant_count        created_at / updated_at
                          status
                          note
                          created_at / updated_at

[user_profiles] 1 ──── * [inquiries]
  id (PK)                 id (PK)
  display_name            customer_profile_id (FK)

                          body

  id (PK)                   id (PK)

                            next_status

                         next_status
                         changed_by (FK)

リレーション 種類 意味
`auth.users` → `user_profiles` 1対1 Supabase Auth の認証主体に対して、アプリ側プロフィールを1件対応させる。
`roles` ↔ `user_profiles` 多対多 `user_role_assignments` を介して、将来の複数権限付与に対応する。
`content_assets` → `top_hero_items` 1対多 1つのアセットを複数のヒーロー/掲載要素から再利用できる。
`user_profiles` → `top_hero_items` / `journey_steps` 1対多 誰が更新したかを保持し、運用履歴を追跡できる。
`user_profiles` → `reservations` / `inquiries` 1対多 1人の顧客が複数の予約・問い合わせを持てる。
`stores` → `reservations` 1対多 1店舗に複数予約が紐づく。店舗管理機能拡張時も整合する。
`reservations` / `inquiries` → 各ステータス履歴 1対多 対応状況の変更履歴を監査目的で残す。

補足: `auth.users` は Supabase 管理テーブルであり、通常の業務カラム追加は `user_profiles` 側で行う。`stores` は現時点では Could 機能だが、予約ログの参照整合性を保つため ER 図に含めている。

2 テーブル定義

ER図に基づき主要テーブルの定義を整理する 認証本体は Supabase 管理テーブル `auth.users` を利用し、本節ではアプリ側で管理するテーブルを中心に定義する。全テーブルに `created_at`, `updated_at` を持たせ、論理削除が必要なテーブルには `deleted_at` を追加する。

auth.users uq_auth_users_email email UNIQUE ログインIDの一意性確保と認証時検索の高速化
user_profiles uq_user_profiles_auth_user_id auth_user_id UNIQUE `auth.users` との1対1対応を保証するため
user_profiles idx_user_profiles_display_name display_name INDEX ユーザー名検索や候補表示に使用するため
user_profiles idx_user_profiles_account_status account_status INDEX 利用状態での絞り込みに使用するため
roles uq_roles_role_code role_code UNIQUE ロールコードの重複を防ぎ、権限参照を高速化するため
user_role_assignments idx_user_role_assignments_user_profile_id user_profile_id INDEX ユーザーごとのロール取得に使用するため
user_role_assignments idx_user_role_assignments_role_id role_id INDEX ロール単位でのユーザー参照に使用するため
user_role_assignments uq_user_role_assignments_user_role user_profile_id, role_id UNIQUE 同一ユーザーへの同一ロール重複付与を防ぐため

content_assets idx_content_assets_file_type file_type INDEX 画像/音源別の絞り込みに使用するため
top_hero_items uq_top_hero_items_display_order_active display_order UNIQUE（deleted_at IS NULL） 有効なヒーロー項目間で表示順の重複を防ぎ、論理削除後の再利用を可能にするため

journey_steps idx_journey_steps_is_visible_step_no is_visible, step_no INDEX 公開対象の取得と導線順ソートに使用するため
stores idx_stores_is_active is_active INDEX 有効店舗の抽出に使用するため
stores idx_stores_store_name store_name INDEX 店舗名検索に使用するため

reservations idx_reservations_status_reserved_at status, reserved_at INDEX 未対応案件の絞り込みと日時順ソートに使用するため
reservations idx_reservations_reserved_at reserved_at INDEX 予約日時での並び替えに使用するため
inquiries idx_inquiries_customer_profile_id customer_profile_id INDEX ユーザー別問い合わせ履歴の取得に使用するため
inquiries idx_inquiries_assigned_to assigned_to INDEX 担当者別の案件抽出に使用するため
inquiries idx_inquiries_status_created_at status, created_at INDEX 未対応案件の絞り込みと新着順表示に使用するため
reservation_status_logs idx_reservation_status_logs_reservation_id reservation_id INDEX 予約ごとの履歴取得に使用するため
