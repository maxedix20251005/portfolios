ER図 必須
テーブル間のリレーション図（テキストベースまたは画像で記載）

リレーション 種類 意味
`auth.users` → `user_profiles` 1対1 Supabase Auth の認証主体に対して、アプリ側プロフィールを1件対応させる。
`roles` ↔ `user_profiles` 多対多 `user_role_assignments` を介して、将来の複数権限付与に対応する。
`content_assets` → `top_hero_items` 1対多 1つのアセットを複数のヒーロー/掲載要素から再利用できる。
`workshop_plans` → `workshop_plan_inclusions` / `workshop_plan_flow_steps` / `workshop_sessions` 1対多 プラン説明、体験フロー、開催日程をプラン単位で管理する。
`user_profiles` → `top_hero_items` / `journey_steps` 1対多 誰が更新したかを保持し、運用履歴を追跡できる。
`user_profiles` → `bookings` / `enquiries` 1対多 1人の顧客が複数の予約・問い合わせを持てる。
`stores` → `bookings` 1対多 1店舗に複数予約が紐づく。店舗管理機能拡張時も整合する。
`workshop_sessions` → `bookings` 1対多 公開カレンダー上の開催枠に対して予約を紐づけ、時点情報を bookings 側に保持する。
`bookings` / `enquiries` → 各ステータス履歴 1対多 対応状況の変更履歴を監査目的で残す。
`auth.users` → `smart_scent_blends` 1対多 ログイン会員ごとにスマート調香候補を保存・再読込できる。

補足: `auth.users` は Supabase 管理テーブルであり、通常の業務カラム追加は `user_profiles` 側で行う。`workshop_sessions` は公開予約カレンダーと管理画面の両方で参照され、`bookings` はその時点のプラン/連絡先スナップショットも保持する。`smart_scent_blends` は seed 投入ではなく、会員操作によって蓄積されるユーザー所有データである。

2 テーブル定義

ER図に基づき主要テーブルの定義を整理する 認証本体は Supabase 管理テーブル `auth.users` を利用し、本節ではアプリ側で管理するテーブルを中心に定義する。全テーブルに `created_at`, `updated_at` を持たせ、論理削除が必要なテーブルには `deleted_at` を追加する。
2026-03-25 以降の `inim-dx/sql` 反映内容として、ワークショップ予約のマスタ/セッション管理テーブル群と、スマート調香の保存テーブル `smart_scent_blends` を追加した。既存 `bookings` テーブルにも、プラン・セッション・連絡先・見積金額を保持するカラムを拡張している。
